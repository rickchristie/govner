# Plan: Multi-Image Architecture for AI Tool Isolation

## Overview

Currently, ALL AI CLI tools (Claude Code, Copilot, Codex, OpenCode) are
installed as layers in a single barrel image. Updating one tool requires
reinstalling all tools (~5-10 minutes). This plan splits the single barrel
image into a **base image** + **one image per AI tool**, so Docker cache
handles everything naturally — updating one tool only rebuilds that image.

### Architecture Change

**Before (single image):**
```
cooper-barrel-base = debian + Go + Node + Python + Claude + Copilot + Codex + OpenCode
                     ↑ change Codex version → reinstall ALL AI tools
```

**After (multi-image):**
```
cooper-base = debian + Go + Node + Python + bubblewrap + CA cert + entrypoint
  ├── cooper-cli-claude   = base + Claude Code
  ├── cooper-cli-copilot  = base + Copilot CLI
  ├── cooper-cli-codex    = base + Codex CLI
  ├── cooper-cli-opencode = base + OpenCode
  └── cooper-cli-custom   = base + user's Dockerfile (user-managed)
                     ↑ change Codex version → only codex rebuilds (~30s)
```

### Key Design Decisions

1. **One image per AI CLI.** Each AI tool has its own Dockerfile and image.
   `cooper cli claude` launches a barrel from the claude image.

2. **Standard Docker caching.** No volume mount tricks, no installer
   containers, no manifest tracking. Just Dockerfiles and `docker build`.

3. **User-extensible.** Users create `~/.cooper/cli/<name>/Dockerfile` with
   `FROM cooper-base:latest` to add custom AI tools. Cooper discovers them
   automatically.

4. **Per-tool aliases.** Each tool image bakes in only its own auto-approve
   alias. The entrypoint reads `COOPER_CLI_TOOL` and `COOPER_CLI_AUTO_APPROVE`
   env vars to set up the alias dynamically.

### UX Change

```
# Before:
cooper cli              # opens shell with ALL tools

# After:
cooper cli claude       # opens shell with Claude (auto-approved)
cooper cli codex        # opens shell with Codex (auto-approved)
cooper cli my-custom    # opens shell from user's custom image
```

## Work Package 3: Entrypoint Changes

### 3.1 Dynamic Auto-Approve Alias

The entrypoint currently hardcodes all tool aliases. Change to read env vars:

**Replace the static aliases block with:**
```bash
# Cooper: Auto-approve alias (set per tool image via ENV)
if [ -n "$COOPER_CLI_TOOL" ] && [ -n "$COOPER_CLI_AUTO_APPROVE" ]; then
    if ! grep -q "Cooper: Auto-approve alias" /home/user/.bashrc 2>/dev/null; then
        cat >> /home/user/.bashrc << ALIAS
# Cooper: Auto-approve alias
alias $COOPER_CLI_TOOL='$COOPER_CLI_TOOL $COOPER_CLI_AUTO_APPROVE'
ALIAS
    fi
fi
```

This way the entrypoint is shared across all images — only the ENV vars differ.

### 3.2 Remove HasClaudeCode/HasCopilot/HasCodex/HasOpenCode from entrypointData

The `entrypointData` struct no longer needs per-tool booleans for aliases.
It still needs `HasGo` (for PATH) and `HasOpenCode` (for Xvfb setup).

Wait — the Xvfb setup is only needed when OpenCode is installed. But the
entrypoint is baked into the **base** image, which is shared. We need to
make the Xvfb startup conditional on the tool:

```bash
# Start Xvfb if this is an OpenCode barrel
if [ "$COOPER_CLI_TOOL" = "opencode" ]; then
    Xvfb :99 -screen 0 1024x768x24 > /dev/null 2>&1 &
    export DISPLAY=:99.0
    if ! grep -q "Cooper: DISPLAY" /home/user/.bashrc 2>/dev/null; then
        echo '# Cooper: DISPLAY for headless clipboard support' >> /home/user/.bashrc
        echo 'export DISPLAY=:99.0' >> /home/user/.bashrc
    fi
fi
```

### 3.3 Updated entrypointData

```go
type entrypointData struct {
    HasGo      bool
    BridgePort int
}
```

Removed: `HasClaudeCode`, `HasCopilot`, `HasCodex`, `HasOpenCode`.
OpenCode-specific logic moved from compile-time template conditionals to
runtime env var checks (`COOPER_CLI_TOOL`).

### 3.4 PATH in .bashrc

The PATH export remains in the entrypoint, but simplified since tool-specific
paths are no longer needed (each tool installs to standard locations within
its own image):

```bash
export PATH="/home/user/.local/bin:/home/user/.npm-global/bin:/home/user/.opencode/bin{{- if .HasGo}}:/usr/local/go/bin:/go/bin{{- end}}:$PATH"
```

Note: `/home/user/.opencode/bin` is kept unconditionally since it's harmless
if the directory doesn't exist, and avoids needing to know at base-image-build
time which tool image will be used.

```go
func ListCLIImages() ([]string, error) {

        "--filter", fmt.Sprintf("reference=%scooper-cli-*", imagePrefix))
```

### 5.1 `runBuild()` Rewrite

```go
func runBuild(cmd *cobra.Command, args []string) error {
    cfg, cooperDir, err := loadConfig()

    // 1. Resolve tool versions.
    resolveLatestVersions(cfg)

    }

    // 3. Stage CA cert, write proxy templates (unchanged).
    ...

    // 4. Build proxy image (unchanged).
    ...

    // 5. Build base image (no AI tools, fast).
    fmt.Fprintln(os.Stderr, "Building base image...")
    docker.BuildImage(docker.GetImageBase(), baseDockerfile, baseDir, uidGidArgs, true)

    // 6. Build each enabled AI tool image.
    for _, tool := range cfg.AITools {
        if !tool.Enabled { continue }
        toolDir := filepath.Join(cliDir, tool.Name)
        imageName := docker.GetImageCLI(tool.Name)
        dockerfile := filepath.Join(toolDir, "Dockerfile")
        fmt.Fprintf(os.Stderr, "Building %s image...\n", tool.Name)
        docker.BuildImage(imageName, dockerfile, toolDir, nil, true)
    }

    // 7. Build user-custom images.
    // Scan cli/ for directories not matching built-in tool names.
    entries, _ := os.ReadDir(cliDir)
    builtinNames := map[string]bool{"claude":true, "copilot":true, "codex":true, "opencode":true}
    for _, e := range entries {
        if !e.IsDir() || builtinNames[e.Name()] { continue }
        customDir := filepath.Join(cliDir, e.Name())
        if fileExists(filepath.Join(customDir, "Dockerfile")) {
            imageName := docker.GetImageCLI(e.Name())
            fmt.Fprintf(os.Stderr, "Building custom image %s...\n", e.Name())
            docker.BuildImage(imageName, filepath.Join(customDir, "Dockerfile"), customDir, nil, true)
        }
    }

    // 8. Update ContainerVersion and save config.
    updateContainerVersions(cfg)
    config.SaveConfig(configPath, cfg)
}
```

### 5.2 First Build vs Subsequent Builds

- **First build:** ~3-5 min (base + all AI tools from scratch)
- **Subsequent builds (nothing changed):** ~5-10 seconds (all cached)
- **One AI tool version changed:** ~30s-2min (only that tool rebuilds)
- **Programming tool changed:** ~2-3 min (base rebuilds, all tool images
  must rebuild since FROM changed)

### 5.3 Remove `CACHE_BUST_LANG` and `CACHE_BUST_AI`

These ARGs are no longer needed. Docker's native caching handles everything:
- If the Dockerfile didn't change, it's cached.
- If the version arg changed, the RUN layer is busted naturally.

For `cooper build`, use `noCache: false` (allow caching) instead of the
current `noCache: true`. This is a key change — `cooper build` should be
idempotent and fast when nothing changed.

The `--no-cache` option should only be used when the user explicitly
requests a clean rebuild (e.g., `cooper build --clean`).

### 5.4 New Flag: `--clean`

```go
buildCmd.Flags().BoolVar(&buildClean, "clean", false, "Force clean rebuild (ignore Docker cache)")
```

When `--clean` is set, pass `noCache: true` to all `BuildImage` calls.
Otherwise, let Docker cache do its job.

## Work Package 6: Update Command Changes

### 6.1 `runUpdate()` Rewrite

The update command becomes much simpler — it only rebuilds images whose
Dockerfile content changed:

```go
func runUpdate(cmd *cobra.Command, args []string) error {
    cfg, cooperDir, err := loadConfig()

    baseChanged := false
    toolsChanged := map[string]bool{}

    // 1. Check programming tools for mismatches.
    for i, tool := range cfg.ProgrammingTools {
        if !tool.Enabled { continue }
        // ... same version detection as current ...
        if mismatch {
            baseChanged = true
            // update PinnedVersion/HostVersion
        }
    }

    // 2. Check AI tools for mismatches.
    for i, tool := range cfg.AITools {
        if !tool.Enabled { continue }
        // ... same version detection as current ...
        if mismatch {
            toolsChanged[tool.Name] = true
            // update PinnedVersion/HostVersion
        }
    }

    if !baseChanged && len(toolsChanged) == 0 {
        fmt.Fprintln(os.Stderr, "All versions match. No rebuild needed.")
        return nil
    }

    // 3. Regenerate templates for changed tools.
    if baseChanged {
        templates.WriteBaseTemplates(baseDir, cfg)
    }
    for name := range toolsChanged {
        templates.WriteCLIToolDockerfile(toolDir, cfg, name)
    }

    // 4. Rebuild base if programming tools changed.
    if baseChanged {
        fmt.Fprintln(os.Stderr, "Rebuilding base image...")
        docker.BuildImage(docker.GetImageBase(), ...)
    }

    // 5. Rebuild only changed tool images.
    // If base changed, ALL tool images need rebuilding (FROM changed).
    // If only specific tools changed, only those images rebuild.
    if baseChanged {
        // Rebuild all enabled tool images
        for _, tool := range cfg.AITools {
            if !tool.Enabled { continue }
            docker.BuildImage(docker.GetImageCLI(tool.Name), ...)
        }
    } else {
        // Rebuild only changed tool images
        for name := range toolsChanged {
            docker.BuildImage(docker.GetImageCLI(name), ...)
        }
    }

    // 6. Hot-reload Squid if AI tools changed (domain whitelist).
    if len(toolsChanged) > 0 { ... }

    // 7. Update ContainerVersion and save config.
    ...
}
```

### 6.2 Key Improvement

With the current single-image approach, `cooper update` when codex changes:
- Regenerates the monolithic Dockerfile
- Rebuilds with `CACHE_BUST_AI` → reinstalls ALL 4 AI tools

With multi-image, `cooper update` when codex changes:
- Regenerates only `~/.cooper/cli/codex/Dockerfile`
- Rebuilds only `cooper-cli-codex` → reinstalls ONLY codex
- Claude, Copilot, OpenCode images are untouched

## Work Package 8: Custom User Images

### 8.1 Discovery

When Cooper lists available CLI tools (for `cooper cli`, configure wizard,
TUI), it:

1. Lists built-in tools from config (`cfg.AITools`)
2. Scans `~/.cooper/cli/` for directories with a `Dockerfile`
3. Directories not matching built-in names are "custom"

```go
func DiscoverCLITools(cfg *config.Config, cooperDir string) []CLITool {
    var tools []CLITool

    // Built-in tools (from config).
    for _, t := range cfg.AITools {
        if !t.Enabled { continue }
        tools = append(tools, CLITool{
            Name:      t.Name,
            IsCustom:  false,
            HasImage:  imageExists(GetImageCLI(t.Name)),
        })
    }

    // Custom tools (from filesystem).
    cliDir := filepath.Join(cooperDir, "cli")
    entries, _ := os.ReadDir(cliDir)
    for _, e := range entries {
        if !e.IsDir() { continue }
        if isBuiltinTool(e.Name()) { continue }
        if fileExists(filepath.Join(cliDir, e.Name(), "Dockerfile")) {
            tools = append(tools, CLITool{

        }
    }

}
```

## Verified Assumptions

All assumptions below were tested in Docker containers on 2026-04-02.

### V1. Docker Layer Sharing Between Child Images — VERIFIED

**Test:** Built `test-base`, `test-childA`, `test-childB` where children use
`FROM test-base`. Inspected layer hashes.

**Result:** All child images share the exact same base layer hashes
(content-addressable dedup). Disk usage is: base layers × 1 + unique child
layer × N. No duplication of base layers.

### V2. Changing One Child Does Not Affect Siblings — VERIFIED

**Test:** Modified `test-childA` Dockerfile and rebuilt. Checked
`test-childB` creation timestamp.

**Result:** `test-childB` was completely untouched — its creation timestamp
didn't change. Rebuild of childA took 0.3s (base layers fully cached).

### V3. Base Change Requires Explicit Child Rebuilds — VERIFIED

**Test:** Modified base image, rebuilt base, then rebuilt children.

**Result:** Children must be explicitly rebuilt (`docker build` again) to
pick up the new base. But rebuilding is fast because unchanged layers within
the base remain cached. Children correctly pick up new base content after
rebuild.

**Implication for Cooper:** `cooper update` must rebuild ALL tool images when
the base changes. This is correct — programming tool changes are rare but
when they happen, all tool images get the new base.

### V4. Subdirectory Dockerfiles With Parent Context — VERIFIED

**Test:** Built from `cli/claude/Dockerfile` and `cli/codex/Dockerfile` using
parent directory as build context.

**Result:** Works perfectly with `-f cli/claude/Dockerfile .` pattern. Each
Dockerfile can reference files in its own directory.

### V5. Multiple Containers Sharing Same Workspace Mount — VERIFIED

**Test:** Started two containers from different images, both mounting
`/tmp/workspace:rw`. One container wrote a file, other container read it.

**Result:** Full bidirectional read-write access. Changes are immediately
visible cross-container.

**Note:** File ownership follows the container's user UID. Since all Cooper
images use the same UID/GID (from base image), ownership is consistent.

### V6. ENV Vars Visible in `docker exec` — VERIFIED

**Test:** Built images with `ENV COOPER_CLI_TOOL=claude`, started container,
ran `docker exec ... env | grep COOPER`.

**Result:** ENV vars set in Dockerfile are visible to `docker exec` commands.
The entrypoint can read `COOPER_CLI_TOOL` and `COOPER_CLI_AUTO_APPROVE` to
set up tool-specific behavior.

### V7. Aliases Work for Interactive Sessions, Not Non-Interactive — VERIFIED

**Test:** Built a realistic image with alias in `.bashrc`. Tested three
execution modes via `docker exec`:

| Mode | Command | Alias works? |
|------|---------|-------------|
| Interactive | `docker exec -i bash -i -c 'alias codex'` | **YES** — `alias codex='codex --full-auto'` |
| Login + Interactive | `docker exec -i bash -l -i -c 'alias codex'` | **YES** — same |
| Non-interactive | `docker exec bash -c 'source ~/.bashrc && alias codex'` | **NO** — `.bashrc` guard (`case $- in *i*)`) returns early |

**Root cause:** Debian's default `.bashrc` has an interactive-mode guard at
the top (`case $- in *i*) ;; *) return;; esac`). In non-interactive shells,
`.bashrc` exits immediately, so aliases appended at the end are never read.

**Implication for Cooper:** This is fine. The alias approach works for the
primary use case:
- `cooper cli codex` → `docker exec -it bash -l` → interactive → alias works ✓
- `cooper cli codex -c "go test"` → non-interactive → alias not set, but
  the one-shot command is `go test`, not `codex ...` — irrelevant ✓

The only edge case: `cooper cli codex -c "codex ask something"` wouldn't get
auto-approve. This is acceptable — users running AI tools interactively is
the 99% case. No wrapper scripts needed.

### V8. Docker Build Caching With No Changes — VERIFIED

**Test:** Built an image, then immediately rebuilt with same Dockerfile.

**Result:** Second build took <1 second. All layers cached. This confirms
that `cooper build` (with `noCache: false`) will be fast when nothing changed.

### V9. Claude Code Install Layer Caching — VERIFIED

## Migration for Existing Users

### First Run After Update

When a user runs `cooper build` with the new code:

1. Old images (`cooper-barrel-base`, `cooper-barrel`) become orphaned.
2. New images (`cooper-base`, `cooper-cli-claude`, etc.) are built from scratch.
3. `cooper cli` now requires a tool name argument.

### Recommended Migration Steps

```
cooper cleanup          # Remove old images
cooper configure        # Regenerate config (no changes needed)
cooper build            # Build new multi-image set
cooper cli claude       # Use the new CLI syntax
```

### Backward Compatibility

- The `config.json` format doesn't change. `AITools` and `ProgrammingTools`
  arrays are the same.
- The proxy image is unchanged.
- `cooper up` is unchanged (it manages proxy, bridge, ACL — not barrels).
- `cooper configure` is mostly unchanged (same screens, same config).

The only breaking UX change: `cooper cli` → `cooper cli <tool-name>`.
